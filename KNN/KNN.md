# 쇼핑몰 방문자의 "구매 확률" 예측하기 — KNN 이야기

> **한 줄 요약**: 쇼핑몰 방문 행동 로그만 보고 "이 사람이 결제할 확률"을 0~1 점수로 매기는 KNN 모델을 만들고, 수업에서 배운 한계까지 직접 실험으로 검증했다.

> - **데이터**: UCI `online_shoppers_intention.csv` — 세션 12,330개, 컬럼 18개

## 0. KNN 알고리즘 원리 — 수업에서 배운 그대로

평가 기준 *"모델 원리 설명 가능 여부"* 대응 — 결과보다 이해를 먼저 정리한다.

### 0-1. 한 줄 정의 (PDF 3쪽)
> **새로 들어온 데이터와 좌표 공간에서 가장 가까운 K개의 학습 데이터를 찾아, 그 K개의 정답으로부터 다수결(분류) 또는 평균(회귀)을 내서 예측하는 알고리즘.**

핵심은 세 단계:

```
① 모든 거리 계산  →  ② 가장 가까운 K개 선택  →  ③ 다수결/평균으로 분류·예측
```

### 0-2. 게으른 학습 (Lazy Learning)
PDF 4쪽: *"k-NN은 학습을 사전에 하지 않고, 새로운 데이터가 입력되면 그 때 학습 및 분류를 수행함."*

→ `model.fit(X, y)`는 사실 **데이터를 저장만** 한다. 진짜 계산(거리 계산, 이웃 찾기)은 `predict()` 호출 시점에 일어남.
→ 그래서 학습은 0초인데 예측은 느림. (선형회귀·신경망과 정반대)

### 0-3. 거리 계산 (PDF 7쪽)
$$d = \sqrt{(a_1 - b_1)^2 + (a_2 - b_2)^2 + \cdots + (a_n - b_n)^2}$$

본 분석에서는 13차원 피처 공간에서 **Manhattan 거리(`p=1`)**가 최종 선정됨. Euclidean(`p=2`)과 비교 후 CV 결과로 결정.
> Manhattan 거리는 각 축 절댓값 합 ($\sum |a_i - b_i|$) — outlier에 더 강건. 본 데이터의 우편향 분포에 더 맞았던 것으로 보임.

### 0-4. K 선택 원칙 (PDF 4~6쪽)
- **K=1**: 가장 가까운 한 명만 보고 따라가 노이즈에 휘둘림 (PDF 5쪽 그림)
- **K=전체**: 전체 데이터 비율에 좌우돼 둔감해짐
- **적절한 K**: 정답이 없고 **실험적으로 비교**해야 함 (PDF 6쪽)
- **분류에서는 홀수**: 다수결 동률 회피 목적 (PDF 4쪽). 회귀(본 분석)에서는 무관하지만 수업 규칙 따라 K 후보 모두 홀수로 구성.

### 0-5. 분류 vs 회귀 — 본 분석의 변형

| 구분 | PDF 13/21쪽 예제 | 본 분석 |
|---|---|---|
| sklearn 클래스 | `KNeighborsClassifier` | `KNeighborsRegressor` |
| K개 이웃 처리 | **다수결** (가장 많은 라벨) | **평균** (0/1의 평균 = 비율) |
| 출력 | 클래스 라벨 (0 또는 1) | 0~1 사이 실수 |
| 동률 회피 | K는 홀수 필요 | 평균이라 무관 |

**핵심**: PDF 14쪽 `predict_proba()`도 "이웃 K명 중 양성 비율"을 반환 → **Classifier와 Regressor는 본 문제에서 수학적으로 동치**.
회귀로 풀면 출력이 처음부터 연속 확률이라 Brier Score 같은 **확률 보정 지표를 자연스럽게 평가** 가능 → 본 분석 목적("구매 확률 0~1 점수")에 더 부합해서 회귀를 선택.

## 1. 문제 정의 — 왜 KNN으로 구매 확률을 예측하는가

### 1-1. 비즈니스 문제
쇼핑몰을 운영할 때, 모든 방문자에게 똑같이 마케팅하면 비효율적. **누가 살 사람인지 점수를 매겨서 그 점수 기준으로 자원을 배분**하고 싶다.

→ "이 방문자의 구매 확률은 몇 %인가?"를 0~1로 답하는 모델.

### 1-2. "비슷한 사람은 비슷하게 행동한다" — KNN의 직관

> **"나랑 비슷하게 둘러본 사람들이 결국 샀다면, 나도 살 가능성이 높지 않을까?"**

이게 KNN이 하는 일이다.

- 행동(체류시간, 페이지 수, 이탈률 등)이 비슷한 K명의 다른 방문자를 찾음
- 그 K명 중에 **실제 산 사람의 비율**을 그대로 "구매 확률"로 씀
- 예: K=10명 중 7명이 샀다면 → 이 사람이 살 확률 = **70%**

### 1-3. KNN을 고른 이유 3가지

**① 결과가 "확률"로 자연스럽게 떨어진다**
선형회귀는 출력이 음수도 나오고 1을 넘기도 해서 확률로 못 씀. 로지스틱 회귀는 확률이 나오지만 "직선으로 자른다"는 강한 가정. **KNN은 그냥 "이웃의 구매 비율"이라 별 가정 없이 0~1 사이 값이 나옴.**

**② "비슷한 사람" 개념이 비즈니스에 직관적**
"왜 이 사람한테 쿠폰 줬어요?" → "이 분이랑 행동이 가장 비슷한 75명 중 80%가 샀거든요" → 설명 가능.

**③ 함수 모양을 미리 가정하지 않는 비모수적 모델**
"구매 확률 = 0.3×체류시간 + 0.5×페이지 수…" 같은 공식을 미리 정하지 않음. 데이터가 알아서 모양을 만들게 둠 → 복잡한 행동 패턴도 잡아낼 수 있음.

## 2. 데이터와 전처리

### 2-1. 데이터 개요
- 12,330개 세션, 18개 컬럼
- 타겟 `Revenue`: True/False → 1/0 변환. **양성 15.5% (불균형)**
- 피처: 페이지 행동 6개, GA 지표 4개, 세션 메타 3개 등

### 2-2. 전처리 방침과 이유

KNN은 **거리(distance)**로 이웃을 찾는 알고리즘. 입력이 전부 숫자여야 하고 스케일이 통일돼야 함.

| 처리 | 이유 |
|---|---|
| `Revenue`, `Weekend` True/False → 1/0 | KNN은 글자 못 읽음 |
| `Month` Jan→1 … Dec→12 | 월의 순서 의미 보존 |
| `VisitorType` Returning=0/New=1/Other=2 | 새 방문자/재방문자 구분 |
| `OperatingSystems`, `Browser`, `Region`, `TrafficType` **제외** | 단순 ID값 (운영체제 "3번"이 "1번"보다 큰 게 아님). 거리 계산에 노이즈만 됨 |
| `StandardScaler`로 모든 수치 정규화 | 체류시간(0~수천초)과 이탈률(0~1)을 같은 스케일로 맞춤. 안 그러면 큰 숫자가 거리 계산 독점 |
| 8:2 분할 + `stratify=y` | 양성 비율 15.5%를 train/test에 동일 유지 |

**최종 입력 13개 피처**: 페이지 행동(6) + GA 지표(4) + 세션 메타(3)

### 2-3. 전처리에서 정직하게 인정하는 한계

거리 기반 모델은 **인코딩 방식이 거리 의미를 그대로 결정**한다. 두 가지는 엄밀히는 부적절:

| 변수 | 처리 방식 | 문제점 | 더 나은 방법 |
|---|---|---|---|
| `Month` | Jan=1 … Dec=12 | 1월(1)과 12월(12) 거리가 11 → 실제로는 한 달 차이 (**순환 변수**) | $\sin(2\pi m/12), \cos(2\pi m/12)$ |
| `VisitorType` | Returning=0, New=1, Other=2 | 명목형(순서 없음)을 정수로 매핑 → 거리 의미 왜곡 | One-Hot 인코딩 |

→ **7장에서 VisitorType One-Hot을 실제로 실험**해 영향을 확인한다.

## 3. 모델 설계 — K 탐색에서 최종 선정까지

평가 기준 *"수업에서 학습한 기법의 적절한 활용"* 대응 — 단순히 K=7 같은 임의값을 박지 않고 수업에서 배운 K 선택 원칙을 데이터로 검증.

### 3-1. K 후보 — 왜 로그 간격인가
KNN에서 이웃 1명이 차지하는 가중치는 **1/K**. K 변화에 따른 예측 변화는 1/K 변화에 비례한다.

| 구간 | 1/K 변화 | 비고 |
|---|---:|---|
| K=1 → 3 | 1.00 → 0.33 (−67%) | 큰 변화 — 촘촘히 봐야 함 |
| K=11 → 13 | 0.091 → 0.077 (−15%) | 차이 거의 없음 |
| K=51 → 53 | 0.020 → 0.019 (−4%) | 노이즈 수준 |

→ **작은 K는 촘촘히, 큰 K는 로그 간격**으로 12점 샘플링:
```
K_SWEEP = [1, 3, 5, 7, 11, 15, 21, 31, 51, 75, 101, 151]
```
$\sqrt{N_{train}} \approx \sqrt{9{,}864} \approx 99$라는 경험칙도 함께 만족 (K≈100 부근까지 커버 → plateau 확인 가능).

### 3-2. 단일 분할 K 스윕 — 첫 감 잡기
test셋 1회 분할에서 K별 결과 (`weights='uniform'`, `p=2`):
- K=1: AUC 0.7038 — 노이즈에 휘둘려 가장 낮음
- K=11: AUC 0.8522 — 가파른 상승
- K=51: AUC 0.8764 — 첫 plateau 진입
- K=151: AUC 0.8596 — 너무 멀리 봐서 둔감해짐

`pct_zero_proba` (확률 0으로 찍히는 비율): K=1에서 85%였다가 K=51에서 26%로 감소 → **양극화 완화** 확인.

> ⚠ 단, 단일 분할은 운에 흔들림. 5-fold CV로 재검증 필요.

### 3-3. 5-fold 교차검증 — 운에 의한 K 변동 제거

| 탐색 차원 | 후보 |
|---|---|
| `n_neighbors` (K) | 12개 (위 K_SWEEP) |
| `weights` | `uniform`, `distance` |
| `p` (거리 종류) | 1 (Manhattan), 2 (Euclidean) |

→ **12 × 2 × 2 = 48 조합 × 5-fold = 총 240번 학습** (수동 3중 for문으로 직접 구현).
`StratifiedKFold`로 fold마다 양성 15.5% 비율 유지.

### 3-4. 최종 선정

> **K=75, weights='distance', p=1 (Manhattan)** — CV ROC-AUC 최고

**왜 이 조합인가**:
- **K=75**: $\sqrt{N_{train}} \approx 99$ 경험칙의 근방. 너무 작지도 너무 크지도 않은 plateau 위.
- **weights='distance'**: 가까운 이웃에 더 큰 가중치 → 양극화 완화 + 노이즈 둔감.
- **p=1 (Manhattan)**: 우편향 분포가 많은 이 데이터에서 outlier에 더 강건. Euclidean(p=2)의 제곱 항이 outlier를 부풀려 거리 의미를 흐리는 걸 피함.

### 3-5. K 선정의 본질적 한계 — PDF가 미리 지적한 부분

PDF 23쪽: *"명확한 근거 제시가 힘들다 — '그냥 이 데이터셋에는 k값을 이렇게 정하니 좋더라..' 주어진 데이터에 맞게 실험적으로 단순하게 판단할 뿐."*

→ 본 분석의 K=75 선정도 **이론적 근거가 아닌 "이 데이터에서 CV AUC 최고였다"는 실험적 결과**다. 이는 KNN 알고리즘 자체의 한계이지 분석의 결함이 아님. **정직하게 인정하는 부분.**

## 4. 평가 지표 이해 — 그게 정확히 뭔지

평가 기준 *"성능 지표의 적절한 사용"* 대응 — 숫자만 보고하지 말고 지표 자체를 이해해야 함.

### 4-1. 5가지 지표를 고른 이유

**먼저 함정 하나**: 정확도(Accuracy)는 쓰지 않았다.
이유 — 전체에서 실제 구매자는 15.5%뿐. 모델이 "모두 안 산다"고만 찍어도 정확도 84.5%가 나옴. **의미 없는 지표**.

| 지표 | 무엇을 보는가 | 왜 골랐나 |
|---|---|---|
| **ROC-AUC** | 확률 점수가 양/음을 잘 구분하는가 | "확률 점수의 순위가 의미 있는가"의 표준 지표 |
| **PR-AUC** | 불균형 상황에서 양성을 잘 잡아내는가 | 양성이 15.5%로 적을 때 ROC-AUC 낙관 편향 보정 |
| **Precision** | 모델이 "살 거다"라고 한 사람 중 실제 산 비율 | "쿠폰을 헛쓰지 않았는가" — 마케팅 효율 |
| **Recall** | 실제 산 사람 중 모델이 잡아낸 비율 | "놓친 고객은 얼마인가" — 기회 손실 |
| **Brier Score** | 예측 확률값 자체가 얼마나 잘 보정됐는가 (↓) | KNN의 강점 "확률 정직성"을 정량화 |

회귀 모델용 R²·MAE·RMSE는 일부러 뺐다 — 이진 분류 평가에는 어색해서.

### 4-2. ROC-AUC — Receiver Operating Characteristic, Area Under Curve

**유래**: 2차 세계대전 미군 레이더 운용자(**R**eceiver **O**perator)가 적기 신호와 잡음을 얼마나 잘 구분하는지 측정한 데서 출발. 의학·기상·머신러닝으로 옮겨옴.

**계산 방법**:
1. 임계값을 1.0 → 0.0으로 천천히 내림
2. 각 임계값에서 두 값 계산:
   - **TPR (= Recall, 민감도)**: 실제 양성 중 잡아낸 비율
   - **FPR (위양성률)**: 실제 음성 중 잘못 양성으로 본 비율
3. (FPR, TPR) 점을 좌표평면에 찍어 곡선 → 그 곡선 **아래 면적** = ROC-AUC

**값의 의미**:
| AUC | 해석 |
|---:|---|
| 1.0 | 완벽한 분류기 |
| 0.9 ~ 1.0 | 매우 좋음 |
| 0.8 ~ 0.9 | 좋음 (본 분석: **0.8830**) |
| 0.5 | 랜덤 (동전 던지기) |

**확률적 해석**:
> **"임의의 양성 1개와 음성 1개를 뽑았을 때, 모델이 양성에 더 높은 점수를 줄 확률."**

본 분석 0.8830 = 무작위로 구매자/비구매자 한 쌍을 골랐을 때 **88.3% 확률로 구매자에게 더 높은 점수를 매긴다**는 뜻. → **순위 지표(ranking metric)**.

**원래 어디서 쓰이나**: 의학 진단, 신용 평가, 추천 랭킹, 이진 분류 일반.

**한계 — 클래스 불균형 시 낙관적**: 음성이 99%인 데이터에서 FPR이 1%→0.1%로 줄어도 곡선 거의 안 변함 → 실무에서 거의 쓸모없는 모델도 0.9 받을 수 있음. **그래서 PR-AUC 보완 필요.**

### 4-3. PR-AUC — Precision-Recall, Area Under Curve

**유래**: **정보 검색(Information Retrieval)** 분야. "구글이 검색 결과 100개 중 진짜 관련 문서는 몇 개 보여줬나?" 같은 평가에서 출발.

**계산 방법**:
1. 임계값을 1.0 → 0.0으로 내리며
2. 각 임계값에서:
   - **Precision**: 양성이라고 한 것 중 진짜 양성 비율
   - **Recall**: 진짜 양성 중 잡아낸 비율
3. (Recall, Precision) 곡선 아래 면적

**값의 의미**:
- **베이스라인 = 양성 비율** (랜덤 분류기의 PR-AUC). 본 분석은 15.5% → 0.155
- 본 분석 PR-AUC = **0.6192** → 베이스라인 대비 **약 4배** → 양성 잡기 능력이 무작위보다 훨씬 좋음
- ROC-AUC와 달리 "0.5가 랜덤"이 아니라 **데이터마다 베이스라인이 다름**

**원래 어디서 쓰이나**: 검색, 사기 탐지, 희귀 질환 스크리닝, 본 분석 같은 구매 예측.

### 4-4. Recall이 낮으면 좋은 건가 나쁜 건가?

본 분석의 임계값 0.5 기준 Recall = **0.306** → 실제 구매자 100명 중 30명만 잡고 70명은 놓침.

**Recall이 높을 때**: 양성을 거의 다 잡음. "**놓치면 큰일 나는**" 상황에 필수 (암 진단, 사기 탐지). 단, **Precision이 떨어지는 trade-off**.

**Recall이 낮을 때**: Precision이 높음. 본 분석이 이 케이스 — Recall 0.306 대신 **Precision 0.7647**. "**확신 있는 사람만 잡자**"는 보수적 모델. 마케팅 예산 빠듯하면 적합, 모든 잠재 구매자 도달 목표면 부족.

**핵심**: 임계값 0.5는 **한 점일 뿐**. KNN이 0~1 연속 점수를 뱉으니 비즈니스 상황에 맞춰 임계값을 자유롭게 조정 가능. ROC-AUC 0.8830은 그 임계값 어디서든 **점수 순위가 양호하다**는 의미.

| 비즈니스 시나리오 | 우선순위 | 임계값 조정 |
|---|---|---|
| 쿠폰 예산 빠듯, 헛돈 NO | Precision ↑ | 0.5 → 0.7로 **올림** |
| 시즌 세일, 잠재 고객 다 잡자 | Recall ↑ | 0.5 → 0.3으로 **내림** |

### 4-5. 한 줄 비교

| 지표 | 초점 | 베이스라인 | 불균형 데이터에서 |
|---|---|---|---|
| ROC-AUC | 양/음 분리 순위 능력 | 0.5 (고정) | 낙관적 함정 |
| PR-AUC | 양성 잡기 능력 | 양성 비율 (가변) | 더 정직 |

### 5-1. 숫자 성능

| 지표 | 값 | 해석 |
|---|---:|---|
| ROC-AUC | **0.8830** | 임의의 구매/비구매 쌍에서 88.3% 확률로 구매자에게 더 높은 점수 |
| PR-AUC | **0.6192** | 베이스라인 0.155 대비 약 4배 — 양성 잡기 능력 우수 |
| Precision (임계 0.5) | **0.7647** | "살 거다"라고 한 10명 중 7~8명이 실제로 삼 |
| Recall (임계 0.5) | **0.3063** | 실제 구매자 10명 중 3명을 잡음 (보수적 운영점) |
| Brier Score | **0.0892** | 베이스라인 0.131 대비 약 32% 개선 — 확률값이 진짜 빈도와 잘 맞음 |

### 5-2. Calibration — 확률값이 진짜로 의미 있는가

"70% 확률"이라고 한 사람들 중 진짜 70%가 사는가? 확인:

| 예측 확률 구간 | 세션 수 | 실제 구매율 |
|---|---:|---:|
| 0.0 (안 살 거다) | 708 | **0.3%** |
| 0.0 ~ 0.2 | 1,246 | **9.0%** |
| 0.2 ~ 0.4 | 305 | **39.3%** |
| 0.4 ~ 0.6 | 102 | **59.8%** |
| 0.6 ~ 0.8 | 59 | **81.4%** |
| 0.8 ~ 1.0 | 46 | **84.8%** |

→ **예측 확률이 올라갈수록 실제 구매율도 따라 올라간다.** 단조 증가.
→ 모델이 뱉는 0.7은 진짜 "70% 정도 살 사람"이라는 뜻.
→ KNN의 핵심 강점 "확률 정직성"이 데이터로 검증됨.

### 5-3. 상위 15명 — 가장 살 것 같다고 본 사람들

모델이 "이 15명이 가장 살 것 같다"고 한 사람들을 보니
→ **13명이 실제로 샀음 (≈87%)**

전체 평균 구매율이 15.5%인데, 상위 15명에서는 87%.
**약 5.6배 더 정확하게 구매자를 찍어낸 것.**

## 6. 개선 검증 — 한계를 실제로 실험

평가 기준 *"모델 비교 분석"* + *"결과 분석 및 해석"* 대응 — KNN.md 부록에서 짚은 한계를 말로만 적지 않고 **실제로 실험해서 효과 측정**.

세 실험 모두 원본과 동일 KNN 설정(K=75, weights='distance', p=1)으로 학습 → ROC-AUC 변동만 본다.

### 6-1. PageValues 제외 — 데이터 누수 의심 검증 ⭐ 핵심 발견

`PageValues`는 해당 세션 페이지가 만들어낸 평균 거래 가치 → **구매 발생 *이후*에 산출되는 사후 정보일 가능성**.

**결과** (PageValues 한 변수만 제거):

| 지표 | 원본 | 제외 후 | Δ |
|---|---:|---:|---:|
| ROC-AUC | 0.8830 | **0.7352** | **−0.1478** |
| PR-AUC | 0.6192 | 0.3148 | −0.3043 |
| Brier | 0.0892 | 0.1193 | +0.0301 |

→ **판정: 데이터 누수 의심 확정.**

AUC 14.8%포인트 폭락은 단일 변수 하나가 신호를 거의 다 담고 있었다는 뜻. 즉 원본 모델의 0.8830은 **부분적으로 "정답을 미리 본 결과"**일 가능성이 크다.

**실배포 시 시사점**:
- `PageValues`는 거래 발생 후 계산되는 후행 지표일 수 있음 → 실제 예측 시점(거래 *전*)에는 값이 없거나 다른 분포를 보일 위험
- 도메인 정의를 다시 확인하고, 데이터 생성 시점을 명확히 한 뒤 사용 여부 결정 필요
- 단순 변수 추가가 아니라 **데이터 누수 검증의 모범 사례**로 기록

### 6-2. VisitorType One-Hot — 명목형 인코딩 일관성 회복

원본은 Returning=0, New=1, Other=2 정수 매핑 → 거리상 "Other가 New보다 2배 멀다"는 의미 왜곡.
One-Hot으로 풀면 세 카테고리가 서로 직교(거리 √2로 동일) → 명목형 본래 의미 회복.

**결과**:

| 지표 | 원본 | One-Hot | Δ |
|---|---:|---:|---:|
| ROC-AUC | 0.8830 | 0.8837 | +0.0007 |
| PR-AUC | 0.6192 | 0.6197 | +0.0005 |
| Brier | 0.0892 | 0.0894 | +0.0003 |
| 피처 수 | 13 | 15 | +2 |

→ **판정: 거의 무영향**.

이론적으로 더 올바른 인코딩이지만 데이터에서 VisitorType의 신호 자체가 약했던 것. **정직한 보고**: "인코딩 한계는 인정하되, 본 데이터에서는 실용적 영향 없음."

### 6-3. 로그 변환 — 우편향 분포 outlier 영향 완화 ⭐ 최고 개선안

`ProductRelated_Duration`(0~수천 초) 같은 체류시간은 **로그 정규에 가까운 우편향 분포**.
StandardScaler만으로는 평균이 outlier에 끌려가 정규화가 왜곡됨 → `log1p(x) = log(1 + x)` 변환 후 정규화하면 거리 계산에서 outlier 가중치 완화.

**왜도(skewness) > 1 변수 자동 탐지 → 10개 변수 변환**:
```
Informational_Duration(7.58), ProductRelated_Duration(7.26), PageValues(6.38),
Administrative_Duration(5.62), ProductRelated(4.34), Informational(4.04),
SpecialDay(3.30), BounceRates(2.95), ExitRates(2.15), Administrative(1.96)
```

**결과**:

| 지표 | 원본 | 로그 변환 | Δ |
|---|---:|---:|---:|
| ROC-AUC | 0.8830 | **0.9111** | **+0.0282** |
| PR-AUC | 0.6192 | **0.6971** | **+0.0780** |
| Brier | 0.0892 | **0.0777** | **−0.0115** |

→ **판정: 세 지표 모두 의미 있는 개선**. AUC 0.91 진입.

**왜 효과적인가**:
- KNN은 거리 계산이 핵심 → outlier가 거리 의미를 왜곡하는 정도가 큼
- 로그 변환은 큰 값을 압축(`log(1000)=6.9` vs `log(10)=2.3`) → outlier가 평균을 덜 끌어당김
- PDF 26쪽 *"이미지의 경우 데이터가 정규화되어 있지 않거나 (...) 분류 성능이 굉장히 안좋아질 수 있다"* 와 같은 맥락 — **거리 기반 모델에서 분포 정규화의 위력 확인**.

### 6-4. 세 실험 종합 비교

| 실험 | 피처 수 | ROC-AUC | PR-AUC | Brier | ΔAUC | 해석 |
|---|---:|---:|---:|---:|---:|---|
| 원본 (전 변수) | 13 | 0.8830 | 0.6192 | 0.0892 | 0.0000 | 기준선 |
| **로그 변환** | 13 | **0.9111** | **0.6971** | **0.0777** | **+0.0282** | ⭐ 최고 개선 |
| VisitorType One-Hot | 15 | 0.8837 | 0.6197 | 0.0894 | +0.0007 | 거의 무영향 |
| PageValues 제외 | 12 | 0.7352 | 0.3148 | 0.1193 | −0.1478 | ⚠ 누수 검증 성공 |

**해석 포인트** (음수 ΔAUC = 나쁜 게 아님):
- 7-3 양수 ΔAUC → 한계 인정이 효과적인 개선으로 이어진 **모범 사례**
- 7-1 큰 음수 ΔAUC → "AUC 폭락"이 **누수 의심 확정의 증거** (정직성의 신호)
- 7-2 미미 → 이론적 한계는 인정하되 실용적 영향 없음

## 8. 한계와 다음 단계

### A. KNN 알고리즘 자체의 한계 (PDF 23쪽 기반)

1. **속도가 느림 (게으른 학습의 대가)** — `predict()` 시점에 모든 학습 데이터와 거리를 계산. 본 분석은 12,330 세션이라 견딜만하지만 100만 단위면 비현실적.
2. **명확한 근거 제시의 어려움** — K=75가 왜 75인지 이론으로 증명 불가, 실험적 결과뿐.
3. **거리 의미의 스케일·인코딩 민감성** — StandardScaler 없었다면 ProductRelated_Duration이 BounceRates를 완전히 묻어버렸을 것.
4. **차원의 저주** — 차원이 커지면 모든 점 간 거리가 비슷해져 "가까운 이웃" 개념 약화.

### B. 본 분석에서 추가로 짚는 한계

5. **Recall 30.6% — "확신만 잡는" 보수적 모델** — 실제 구매자 70%를 놓침. 모든 잠재 구매자에 도달 목표면 임계값 조정 필요.
6. **`PageValues` 데이터 누수 의심 확인됨** — 7-1 실험으로 확정. 실배포 전 도메인 정의 확인 필수.
7. **카테고리 정보 손실** — OS/Browser/Region/TrafficType 4개 제외 → 세그먼트별 분석 불가.

### C. 다음 단계 (우선순위 순)

1. ✅ **로그 변환 적용** (7-3에서 검증 완료, AUC 0.91 달성) → 본 모델의 기본값으로 채택
2. **PageValues 도메인 검증** (7-1 확인 → 다음 우선순위) → 거래 발생 시점 명확화
3. **Month 순환 인코딩** — `sin/cos` 변환으로 12월-1월 거리 왜곡 해결
4. ✗ **VisitorType One-Hot** — 영향 미미 확인됨 (7-2). 우선순위 하향
5. 버린 카테고리 ID 원-핫 + PCA로 차원 압축 후 재투입
