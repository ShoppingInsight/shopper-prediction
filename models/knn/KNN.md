## 1. 문제 정의 - 누가 구매할 가능성이 높은가

쇼핑몰 운영자는 모든 방문자에게 똑같이 쿠폰이나 광고비를 쓰면 비효율적이다. 그래서 방문자마다 구매 가능성을 점수화하고, 그 점수에 따라 마케팅 자원을 배분하고 싶다.

이번 분석의 목표는 다음 질문에 답하는 것이다.

> **"현재 쇼핑몰에 들어온 방문자가 실제로 구매할 확률은 몇 %인가?"**

따라서 단순히 `구매/미구매`를 맞히는 것보다, 방문자별로 **0~1 사이의 구매 확률 점수**를 만드는 것이 중요하다.

## 3. 전처리 - KNN이 거리 계산을 할 수 있게 만들기

KNN은 각 데이터를 좌표 공간의 한 점으로 보고, 점과 점 사이의 거리를 계산한다. 그래서 입력값은 숫자여야 하고, 변수 간 스케일도 맞아야 한다.

### 3-1. 실제 전처리 방식

| 처리 | 이유 |
|---|---|
| `Revenue`, `Weekend`를 True/False에서 1/0으로 변환 | 모델이 숫자로 계산할 수 있게 하기 위해 |
| `Month`를 Jan=1, Feb=2, ...로 변환 | 월의 순서 정보를 숫자로 표현하기 위해 |
| `VisitorType`을 Returning=0, New=1, Other=2로 변환 | 방문자 유형을 숫자로 표현하기 위해 |
| `OperatingSystems`, `Browser`, `Region`, `TrafficType` 제외 | 단순 ID값이라 거리 계산에 부적절하기 때문에 |
| `StandardScaler`로 수치형 변수 정규화 | 체류 시간처럼 큰 값이 거리 계산을 독점하지 않게 하기 위해 |
| 8:2 train/test 분할, `stratify=y` 적용 | 구매 비율 15.5%를 학습/평가 데이터에 동일하게 유지하기 위해 |

최종 모델에는 **13개 피처**가 들어갔다.

### 3-2. 전처리의 한계도 인정해야 한다

KNN은 거리 기반 모델이기 때문에 인코딩 방식이 곧 거리의 의미가 된다. 그래서 아래 두 처리는 엄밀히 말하면 완벽하지 않다.

| 변수 | 이번 처리 | 문제점 | 더 나은 방법 |
|---|---|---|---|
| `Month` | Jan=1, ..., Dec=12 | 1월과 12월이 멀게 계산된다. 실제로는 한 달 차이다. | `sin`, `cos` 순환 인코딩 |
| `VisitorType` | Returning=0, New=1, Other=2 | 순서가 없는 범주에 순서가 생긴다. | One-Hot 인코딩 |

이 한계는 뒤의 개선 실험에서 실제로 검증했다.

## 4. KNN 알고리즘 원리

KNN은 **K-Nearest Neighbors**, 즉 "가장 가까운 K개의 이웃"을 보는 알고리즘이다.

### 4-1. 한 줄 정의

> 새로 들어온 데이터와 가장 가까운 K개의 학습 데이터를 찾고, 그 이웃들의 정답을 이용해 예측하는 알고리즘.

작동 순서는 단순하다.

```text
1. 새 데이터와 모든 학습 데이터 사이의 거리를 계산한다.
2. 거리가 가장 가까운 K개 데이터를 고른다.
3. 분류 문제면 다수결, 회귀 문제면 평균으로 예측한다.
```

### 4-2. 게으른 학습, Lazy Learning

KNN은 일반적인 모델처럼 학습 단계에서 복잡한 공식을 만들지 않는다.

`model.fit(X, y)`에서는 데이터를 거의 저장만 하고, 실제 거리 계산은 `predict()`할 때 일어난다.

그래서 KNN은 다음 특징을 가진다.

| 단계 | 특징 |
|---|---|
| 학습 | 빠르다. 데이터를 저장하는 수준이다. |
| 예측 | 느릴 수 있다. 새 데이터가 들어올 때마다 기존 데이터와 거리를 계산해야 한다. |

즉, KNN은 "학습은 게으르게, 예측은 열심히" 하는 모델이다.

### 4-3. 거리 계산

기본적인 유클리드 거리는 다음과 같다.

$$d = \sqrt{(a_1 - b_1)^2 + (a_2 - b_2)^2 + \cdots + (a_n - b_n)^2}$$

이번 최종 모델에서는 유클리드 거리보다 **Manhattan 거리**가 더 좋았다.

$$d = |a_1 - b_1| + |a_2 - b_2| + \cdots + |a_n - b_n|$$

Manhattan 거리는 각 변수 차이의 절댓값을 더한다. 이번 데이터처럼 우편향 분포와 이상치가 많은 경우, 제곱으로 차이를 크게 키우는 유클리드 거리보다 더 안정적으로 작동한 것으로 볼 수 있다.

### 4-4. K값 선택의 의미

K는 "몇 명의 이웃을 볼 것인가"를 정하는 값이다.

| K값 | 특징 |
|---|---|
| 너무 작음 | 가장 가까운 몇 명에게만 끌려가서 노이즈에 민감하다. |
| 너무 큼 | 너무 많은 사람을 평균내서 개별 패턴을 놓친다. |
| 적절한 K | 노이즈와 일반화 사이의 균형을 잡는다. |

K에는 정답이 없다. 그래서 이번 분석에서도 여러 K를 직접 비교하고, 교차검증으로 최종 K를 정했다.

## 5. 왜 KNN을 선택했는가

이번 문제에서 KNN의 직관은 매우 자연스럽다.

> **"나와 비슷하게 쇼핑몰을 둘러본 사람들이 많이 구매했다면, 나도 구매할 가능성이 높지 않을까?"**

KNN은 이 직관을 그대로 구현한다.

1. 방문자의 행동 데이터를 좌표로 표현한다.
2. 행동이 가장 비슷한 K명의 과거 방문자를 찾는다.
3. 그 K명 중 실제 구매한 사람의 비율을 구매 확률로 사용한다.

예를 들어 K=10일 때 비슷한 방문자 10명 중 7명이 실제로 구매했다면, 새 방문자의 구매 확률은 70%로 볼 수 있다.

### 5-1. 회귀 모델로 사용한 이유

일반적으로 KNN은 분류 모델로 많이 소개되지만, 이번 분석에서는 `KNeighborsRegressor`를 사용했다.

| 구분 | KNN 분류 | 이번 분석의 KNN 회귀 |
|---|---|---|
| sklearn 클래스 | `KNeighborsClassifier` | `KNeighborsRegressor` |
| 이웃 처리 | 다수결 | 평균 |
| 출력 | 0 또는 1 | 0~1 사이 실수 |
| 해석 | 구매/미구매 라벨 | 구매 확률 점수 |

타겟을 구매=1, 미구매=0으로 두면, KNN 회귀의 평균값은 곧 "이웃 중 구매자 비율"이 된다.

즉, 이번 문제에서는 회귀 출력이 그대로 확률처럼 해석된다. 그래서 Brier Score처럼 확률값 자체의 품질을 평가하는 지표도 자연스럽게 사용할 수 있다.

## 6. 모델 설계 - K와 거리 기준 정하기

처음부터 K=75를 정한 것이 아니라, 작은 K부터 큰 K까지 비교했다.

### 6-1. K 후보를 로그 간격으로 잡은 이유

KNN에서 이웃 한 명이 차지하는 영향은 대략 `1/K`다. 작은 K에서는 K가 조금만 바뀌어도 영향이 크게 변하지만, 큰 K에서는 변화가 작다.

| 구간 | 1/K 변화 | 의미 |
|---|---:|---|
| K=1 -> 3 | 1.00 -> 0.33 | 변화가 매우 큼 |
| K=11 -> 13 | 0.091 -> 0.077 | 변화가 작음 |
| K=51 -> 53 | 0.020 -> 0.019 | 거의 비슷함 |

그래서 작은 K는 촘촘하게, 큰 K는 넓게 보도록 다음 후보를 사용했다.

```python
K_SWEEP = [1, 3, 5, 7, 11, 15, 21, 31, 51, 75, 101, 151]
```

학습 데이터가 약 9,864개이므로 `sqrt(N_train) ≈ 99`라는 경험칙도 함께 고려했다.

### 6-2. 단일 분할에서 먼저 확인한 결과

먼저 test set 한 번 분할에서 K별 성능을 봤다.

> **노트북에서 같이 볼 부분**: `## 수동 K 스윕 — K마다 직접 학습해서 결과 비교` 출력 표

| K | 결과 |
|---:|---|
| 1 | AUC 0.7038, 노이즈에 매우 민감 |
| 11 | AUC 0.8522, 성능이 크게 상승 |
| 51 | AUC 0.8764, 성능이 안정되는 구간 |
| 151 | AUC 0.8596, 너무 많은 이웃을 봐서 둔감해짐 |

K가 너무 작으면 예측 확률이 0 또는 1에 몰리는 문제가 있었다. K=1에서는 예측 확률이 0으로 찍히는 비율이 85%였고, K=51에서는 26%까지 줄었다.

대표 출력은 이렇게 읽으면 된다.

```text
K=1   test_AUC=0.7038, pct_zero_proba=0.8471
K=11  test_AUC=0.8522, pct_zero_proba=0.5166
K=51  test_AUC=0.8764, pct_zero_proba=0.2640
K=151 test_AUC=0.8596, pct_zero_proba=0.1594

테스트 AUC 최고: K=51
```

하지만 단일 분할 결과는 운에 흔들릴 수 있으므로, 최종 선택은 5-fold 교차검증으로 했다.

### 6-3. 5-fold 교차검증

최종 탐색은 다음 조합으로 진행했다.

| 탐색 대상 | 후보 |
|---|---|
| `n_neighbors` | 12개 K 후보 |
| `weights` | `uniform`, `distance` |
| `p` | 1(Manhattan), 2(Euclidean) |

총 조합은 `12 × 2 × 2 = 48개`이고, 5-fold 교차검증을 적용했으므로 총 240번 학습했다.

또한 `StratifiedKFold`를 사용해 각 fold마다 구매 비율이 비슷하게 유지되도록 했다.

### 6-4. 최종 선정 모델

교차검증 결과 최종 모델은 다음과 같다.

> **노트북에서 같이 볼 부분**: `## 5-fold 교차검증 — 수동 for문으로 직접 구현` 출력

```text
K = 75
weights = "distance"
p = 1  # Manhattan distance
```

대표 출력:

```text
탐색 조합 수 : 12 × 2 × 2 = 48
각 조합 학습 횟수 : 5 (fold) → 총 240번 학습

CV AUC 상위 5개 조합
K=75,  weights=distance, p=1, cv_AUC=0.8975
K=51,  weights=distance, p=1, cv_AUC=0.8974
K=101, weights=distance, p=1, cv_AUC=0.8962

선정: K=75, weights='distance', p=1
```

이 조합을 고른 이유는 다음과 같다.

| 선택 | 이유 |
|---|---|
| K=75 | 너무 작지도 크지도 않은 안정 구간이며, `sqrt(N)` 경험칙 근처 |
| `weights="distance"` | 가까운 이웃에게 더 큰 가중치를 줘서 거리 정보를 더 잘 활용 |
| `p=1` | 우편향 분포와 이상치가 많은 데이터에서 더 안정적으로 작동 |

단, K=75라는 값은 이론적으로 증명된 정답이 아니다. **이 데이터에서 교차검증 성능이 가장 좋았던 실험적 선택**이다. 이 점은 KNN의 본질적 한계이기도 하다.

## 7. 평가 지표 - 정확도 대신 무엇을 봤는가

이번 데이터는 구매자가 15.5%뿐이므로 정확도는 적절하지 않다. 모두 미구매라고 해도 84.5% 정확도가 나오기 때문이다.

그래서 다음 지표를 사용했다.

| 지표 | 무엇을 보는가 | 선택 이유 |
|---|---|---|
| ROC-AUC | 구매자와 비구매자를 점수 순서로 잘 구분하는가 | 확률 점수의 순위 품질을 보기 위해 |
| PR-AUC | 적은 수의 구매자를 얼마나 잘 잡는가 | 클래스 불균형 상황에 더 정직하기 때문에 |
| Precision | 구매할 것이라고 예측한 사람 중 실제 구매자 비율 | 쿠폰이나 광고비 낭비를 줄이는 관점 |
| Recall | 실제 구매자 중 모델이 잡아낸 비율 | 놓친 구매 기회를 보는 관점 |
| Brier Score | 예측 확률 자체가 실제 빈도와 잘 맞는가 | 확률 점수의 보정 품질을 보기 위해 |

### 7-1. ROC-AUC

ROC-AUC는 임의의 구매자 1명과 비구매자 1명을 뽑았을 때, 모델이 구매자에게 더 높은 점수를 줄 확률로 해석할 수 있다.

| AUC | 해석 |
|---:|---|
| 1.0 | 완벽 |
| 0.9 이상 | 매우 좋음 |
| 0.8~0.9 | 좋음 |
| 0.5 | 랜덤 |

이번 원본 모델의 ROC-AUC는 **0.8830**이다. 즉, 구매자와 비구매자 한 쌍을 무작위로 뽑았을 때 약 88.3% 확률로 구매자에게 더 높은 점수를 준다는 뜻이다.

다만 ROC-AUC는 불균형 데이터에서 낙관적으로 보일 수 있다. 그래서 PR-AUC를 함께 봤다.

### 7-2. PR-AUC

PR-AUC는 Precision과 Recall의 관계를 보는 지표다. 검색, 사기 탐지, 희귀 질환 예측처럼 양성이 적은 문제에서 중요하다.

PR-AUC의 랜덤 기준선은 0.5가 아니라 **양성 비율**이다. 이번 데이터의 양성 비율은 15.5%이므로 기준선은 약 0.155다.

원본 모델의 PR-AUC는 **0.6192**로, 기준선보다 약 4배 높다. 즉, 구매자를 잡아내는 능력이 무작위보다 훨씬 좋다.

### 7-3. Precision과 Recall

원본 모델에서 임계값 0.5를 사용하면 다음과 같다.

| 지표 | 값 | 해석 |
|---|---:|---|
| Precision | 0.8047 | 구매할 것이라고 예측한 10명 중 약 8명이 실제 구매 |
| Recall | 0.2696 | 실제 구매자 10명 중 약 3명만 잡음 |

이 모델은 많은 구매자를 넓게 잡는 모델이라기보다, **확신 있는 사람만 골라내는 보수적인 모델**이다.

비즈니스 목표에 따라 임계값은 조정할 수 있다.

| 상황 | 목표 | 임계값 |
|---|---|---|
| 쿠폰 예산이 적다 | Precision 높이기 | 0.5보다 올림 |
| 많은 잠재 고객을 잡고 싶다 | Recall 높이기 | 0.5보다 내림 |

### 8-1. 확률 보정 확인

KNN이 "70% 구매 확률"이라고 말했을 때, 실제로 그 정도 비율이 구매하는지도 확인했다.

| 예측 확률 구간 | 세션 수 | 실제 구매율 |

| 0.6~0.8 | 60 | 85.0% |
| 0.8~1.0 | 37 | 83.8% |

예측 확률이 높아질수록 실제 구매율도 함께 높아진다. 따라서 모델의 점수는 단순한 순위뿐 아니라 확률값으로도 어느 정도 의미가 있다.

### 8-2. 상위 15명 확인

모델이 가장 구매 가능성이 높다고 본 상위 15명을 확인했을 때, 그중 **13명이 실제 구매자**였다.

발표 때는 노트북의 `▼ 예측 확률 상위 15` 표를 보여주면 된다. 표에서 `actual_revenue`가 1이면 실제 구매, 0이면 미구매다. 상위 15명 중 0은 2명뿐이고 나머지 13명은 구매자다.

## 9. 개선 및 검증 실험

여기서는 원본 모델의 한계를 말로만 지적하지 않고, 실제 실험으로 확인했다. 모든 실험은 원본과 같은 KNN 설정인 `K=75`, `weights="distance"`, `p=1`을 사용했다.

> **노트북에서 같이 볼 부분**: `# 7. 개선 검증 — KNN.md 부록 C에서 짚은 한계 항목 실험`

### 9-1. PageValues 제외 - 데이터 누수 의심 검증

`PageValues`는 해당 세션의 페이지가 만들어낸 평균 거래 가치와 관련된 변수다. 문제는 이 값이 구매 이후에 계산되는 사후 정보일 가능성이 있다는 점이다.

만약 예측 시점보다 나중에 알 수 있는 정보가 모델에 들어가면, 모델은 미래 정보를 미리 본 것이 된다. 이것을 **데이터 누수**라고 한다.

`PageValues` 하나만 제거한 결과는 다음과 같다.

| 지표 | 원본 | PageValues 제외 | 변화 |
|---|---:|---:|---:|
| ROC-AUC | 0.8830 | **0.7352** | -0.1478 |
| PR-AUC | 0.6192 | 0.3148 | -0.3043 |
| Brier Score | 0.0892 | 0.1193 | +0.0301 |
| Precision | 0.8047 | 0.2500 | -0.5547 |
| Recall | 0.2696 | 0.0052 | -0.2644 |
| 상위 15명 실제 구매자 | 13명 | 7명 | -6명 |

결과적으로 AUC가 크게 떨어졌다. 이는 `PageValues` 하나가 모델 성능에 매우 큰 영향을 주고 있었다는 뜻이다.

따라서 원본 모델의 높은 성능은 일부 `PageValues`에 의존했을 수 있으며, 실제 배포 전에는 이 변수가 예측 시점에 사용 가능한 정보인지 반드시 확인해야 한다.

이 결과는 "PageValues를 빼면 모델이 망했다"가 아니라, **PageValues 없이 기대할 수 있는 더 정직한 실전 성능은 AUC 0.7352 수준**이라는 뜻으로 해석하는 게 맞다. 랜덤 기준 0.5보다는 높으므로 구매 가능성 순위 신호는 남아 있지만, 원본처럼 강한 타겟팅 모델이라고 말하기는 어렵다.

### 9-2. VisitorType One-Hot - 범주형 인코딩 검증

원본에서는 `VisitorType`을 Returning=0, New=1, Other=2로 처리했다. 하지만 이 변수는 순서가 없는 범주형 변수이므로 One-Hot 인코딩이 더 자연스럽다.

실험 결과는 다음과 같다.

| 지표 | 원본 | One-Hot | 변화 |
|---|---:|---:|---:|
| ROC-AUC | 0.8830 | 0.8837 | +0.0007 |
| PR-AUC | 0.6192 | 0.6197 | +0.0005 |
| Brier Score | 0.0892 | 0.0894 | +0.0003 |
| 피처 수 | 13 | 15 | +2 |

이론적으로는 One-Hot이 더 올바르지만, 이번 데이터에서는 성능 차이가 거의 없었다. 즉, `VisitorType`의 인코딩 문제는 인정하되 실용적 영향은 작았다.

### 9-3. 로그 변환 - 우편향 분포 완화

KNN은 거리 계산이 핵심이므로 이상치와 우편향 분포에 민감하다. 예를 들어 체류 시간이 몇 초인 사람과 수천 초인 사람이 함께 있으면, 큰 값이 거리 계산을 과도하게 지배할 수 있다.

그래서 왜도(skewness)가 1보다 큰 변수에 `log1p(x) = log(1+x)` 변환을 적용했다.

로그 변환 대상 변수는 다음과 같다.

```text
Informational_Duration, ProductRelated_Duration, PageValues,
Administrative_Duration, ProductRelated, Informational,
SpecialDay, BounceRates, ExitRates, Administrative
```

결과는 가장 좋았다.

| 지표 | 원본 | 로그 변환 | 변화 |
|---|---:|---:|---:|
| ROC-AUC | 0.8830 | **0.9111** | **+0.0282** |
| PR-AUC | 0.6192 | **0.6971** | **+0.0780** |
| Brier Score | 0.0892 | **0.0777** | **-0.0115** |

로그 변환은 큰 값을 압축해서 이상치의 영향을 줄인다. 그 결과 거리 계산이 더 안정화됐고, 세 지표가 모두 개선됐다.

### 9-4. 실험 종합

| 실험 | 피처 수 | ROC-AUC | PR-AUC | Brier Score | AUC 변화 | 해석 |
|---|---:|---:|---:|---:|---:|---|
| 원본 | 13 | 0.8830 | 0.6192 | 0.0892 | 0.0000 | 기준선 |
| 로그 변환 | 13 | **0.9111** | **0.6971** | **0.0777** | **+0.0282** | 최고 개선 |
| VisitorType One-Hot | 15 | 0.8837 | 0.6197 | 0.0894 | +0.0007 | 거의 영향 없음 |
| PageValues 제외 | 12 | 0.7352 | 0.3148 | 0.1193 | -0.1478 | 데이터 누수 의심 확인 |

가장 중요한 발견은 두 가지다.

1. **로그 변환은 KNN에 매우 효과적이었다.** 거리 기반 모델에서는 분포 정리가 성능에 직접 영향을 준다.
2. **PageValues는 데이터 누수 가능성이 크다.** 성능이 좋더라도 예측 시점에 쓸 수 없는 정보라면 실제 서비스에는 사용할 수 없다.

### 9-5. 최종 인사이트

이번 분석의 핵심 인사이트는 다음과 같다.

> **방문 행동 데이터만으로도 구매 가능성을 어느 정도 순위화할 수 있지만, 높은 성능의 상당 부분은 `PageValues`에 의존했다. 따라서 이 모델의 진짜 결론은 "KNN이 구매자를 잘 맞혔다"가 아니라, "마케팅 타겟팅 가능성은 보이지만 데이터 누수 검증 없이는 높은 성능을 믿으면 안 된다"이다.**

발표에서는 이 내용을 세 문장으로 정리하면 된다.

1. `PageValues`를 포함하면 ROC-AUC 0.8830으로 구매 가능성이 높은 고객을 잘 골라낸다.
2. 하지만 `PageValues`를 제거하면 ROC-AUC가 0.7352로 떨어져, 원본 성능이 해당 변수에 크게 의존했음을 확인했다.
3. 그러므로 실무 적용 전에는 `PageValues`가 예측 시점에 사용 가능한 변수인지 확인하고, 사용할 수 없다면 AUC 0.7352를 더 현실적인 기준 성능으로 봐야 한다.

## 10. 비즈니스 활용

모델의 구매 확률 점수는 마케팅 우선순위를 정하는 데 사용할 수 있다. 단, 활용 시나리오는 `PageValues` 사용 가능 여부에 따라 나뉜다.

| 시나리오 | 해석 | 활용 |
|---|---|---|
| `PageValues` 사용 가능 | AUC 0.8830. 구매 가능성이 높은 고객을 꽤 잘 선별 | 강한 타겟팅 모델로 활용 가능 |
| `PageValues` 사용 불가 | AUC 0.7352. 신호는 남지만 선별력은 중간 수준 | 보조 점수나 추가 모델 개선의 기준선으로 활용 |

| 예측 확률대 | 의미 | 전략 |
|---|---|---|
| 0.0~0.2 | 구매 가능성이 낮은 방문자 | 광고비나 쿠폰을 아낀다. |
| 0.4~0.8 | 설득하면 구매할 수 있는 방문자 | 쿠폰, 리타게팅, 장바구니 알림을 집중한다. |
| 0.8 이상 | 이미 구매 가능성이 높은 방문자 | 할인보다 번들, 업셀 전략을 쓴다. |

특히 모델 상위 15명 중 13명이 실제 구매자였다는 점은, 한정된 마케팅 예산을 누구에게 먼저 쓸지 정하는 데 모델이 도움을 줄 수 있음을 보여준다.

## 11. 한계와 다음 단계

### 11-1. KNN 자체의 한계

| 한계 | 설명 |
|---|---|
| 예측 속도가 느릴 수 있음 | 새 데이터마다 기존 데이터와 거리를 계산해야 한다. |
| K 선택의 이론적 근거가 약함 | K=75는 교차검증 결과이지 수학적으로 증명된 정답은 아니다. |
| 스케일과 인코딩에 민감함 | 정규화나 범주형 처리 방식에 따라 거리 의미가 달라진다. |
| 차원의 저주 | 피처가 너무 많아지면 가까운 이웃이라는 개념이 약해진다. |

### 11-2. 이번 분석의 한계

| 한계 | 설명 |
|---|---|
| Recall이 낮음 | 임계값 0.5에서는 실제 구매자 중 약 27.0%만 잡는다. |
| PageValues 누수 가능성 | 실제 예측 시점에 사용할 수 있는 변수인지 확인이 필요하다. |
| 일부 카테고리 변수 제외 | OS, Browser, Region, TrafficType을 제외해 세그먼트 정보가 빠졌다. |
| Month 인코딩 | 12월과 1월이 멀게 계산되는 문제가 남아 있다. |

### 11-3. 다음 단계 우선순위

1. **로그 변환을 기본 전처리로 채택한다.** 이미 AUC 0.9111로 개선 효과가 확인됐다.
2. **PageValues의 생성 시점을 확인한다.** 예측 시점에 쓸 수 없다면 제거하거나 별도 모델을 만들어야 한다.
3. **Month를 순환 인코딩한다.** 12월과 1월의 거리 왜곡을 줄일 수 있다.
4. **버린 카테고리 변수를 One-Hot 또는 차원 축소와 함께 다시 실험한다.**
5. **비즈니스 목적에 맞춰 임계값을 조정한다.** 예산 절감이면 Precision, 고객 확보면 Recall 중심으로 조정한다.

## 부록. 재현 방법

`purchase_proba.ipynb`를 위에서부터 순서대로 실행하면 주요 수치를 재현할 수 있다.

실행 흐름은 다음과 같다.

1. CSV 로드
2. Boolean, Month, VisitorType 전처리
3. train/test 분할 및 정규화
4. 초기 KNN 모델 학습
5. K 후보 스윕
6. 5-fold 교차검증으로 최종 모델 선정
7. ROC-AUC, PR-AUC, Precision, Recall, Brier Score 계산
8. Calibration과 상위 예측 결과 확인
9. PageValues 제외, VisitorType One-Hot, 로그 변환 실험
10. 실험 결과 종합 비교
