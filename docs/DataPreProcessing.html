온라인 쇼핑 세션 데이터 전처리

이 문서는 online_shoppers_intention.csv를 모델 학습 전에 어떻게 가공하는지 설명한다. 핵심은 공통 전처리를 한 번만 수행하고, 이후에는 KNN과 Tree 계열 모델의 특성에 맞춰 서로 다른 입력 데이터를 만드는 것이다.

1. 전체 흐름

전처리는 결과 비교가 아니라 학습에 들어갈 데이터를 만드는 과정이다. 따라서 성능 수치보다 “왜 이 변환이 이 모델에 필요한가”를 기준으로 읽으면 된다.

원본 CSV 12,330개 세션과 18개 컬럼을 로드한다.
공통 확인 결측치, 타깃 분포, 데이터 타입을 확인한다.
공통 split 모든 모델이 같은 train/test 쌍을 사용한다.
KNN 분기 거리 계산에 맞게 차원과 스케일을 정리한다.
Tree 분기 분할 모델에 맞게 정보를 최대한 보존한다.

2. 데이터와 공통 전처리

12,330 세션 수
18 타깃 포함 컬럼 수
15.47% Revenue=True 비율
0 결측 셀 수

공통으로 하는 일

단계 처리 이유
로드 pd.read_csv('../data/online_shoppers_intention.csv') 원본 데이터를 한 번만 읽고 이후 모델별 복사본을 만든다.
결측치 확인 df.isnull().sum() 결측치가 없으므로 imputation 단계는 생략한다.
타깃 변환 Revenue를 Boolean에서 0/1 정수로 변환 모델과 평가 코드가 다루기 쉬운 형태로 맞춘다.
분할 train_test_split(..., stratify=y) 학습셋과 평가셋의 구매 비율을 비슷하게 유지한다.
누수 방지 원칙: 스케일러, 인코더처럼 fit이 필요한 변환은 train 데이터로만 학습하고 test에는 transform만 적용한다. 그래서 공통 단계는 split까지만 두고, 모델별 전처리는 split 이후에 수행한다. 
y_full = df['Revenue'].astype(int)
X_full = df.drop(columns=['Revenue'])

X_train, X_test, y_train, y_test = train_test_split(
    X_full, y_full,
    test_size=0.2,
    random_state=42,
    stratify=y_full
)

3. 왜 모델별 전처리가 달라지는가

KNN 거리 기반 모델

KNN은 각 행을 좌표 공간의 점으로 보고 가까운 이웃을 찾는다. 숫자 크기, 단위, 차원 수가 그대로 거리 계산에 들어가므로 전처리 영향이 크다.

  • 스케일이 큰 변수가 거리를 지배할 수 있다.
  • ID처럼 의미 없는 숫자도 거리로 계산될 수 있다.
  • 차원이 늘면 이웃 개념이 약해질 수 있다.

Tree 분할 기반 모델

Decision Tree와 Random Forest는 변수별 임계값으로 데이터를 나눈다. 숫자의 절대 크기보다 정렬 순서와 분할 가능성이 중요하다.

  • 스케일링이나 log 변환이 필수는 아니다.
  • 차원이 늘어도 무용 변수는 정보 이득으로 덜 쓰인다.
  • 범주형 변수는 One-Hot으로 분리해두면 해석이 쉽다.

4. KNN 전처리 파이프라인

KNN의 목표는 거리 의미를 깨뜨리는 변수를 줄이고, 남은 변수의 단위를 맞추는 것이다. 노트북에서는 X_tr_knn, X_te_knn 복사본을 만들어 이 흐름을 실행한다.

4-1. 컬럼 선택

처리 컬럼 판단 이유
제거 Month 월을 숫자로 두면 1월과 12월이 멀게 계산되는 등 거리 의미가 부자연스럽다.
제거 OperatingSystems, Browser, Region, TrafficType ID 성격의 정수라 숫자 차이가 실제 유사도 차이를 뜻하지 않는다.
압축 VisitorType → is_new_visitor One-Hot으로 차원을 늘리는 대신 신규 방문 여부 한 비트만 남긴다.
제거 PageValues 세션 종료 후 계산되는 값일 가능성이 높아 예측 시점 누수로 판단한다.

4-2. PageValues는 왜 빼는가

PageValues는 Google Analytics 계열 지표로, 거래가 발생한 뒤 페이지 가치가 역산되는 구조다. 즉 예측 시점에 이미 알고 있다고 가정하면 구매 결과를 미리 본 것과 비슷해질 수 있다.

문서와 노트북에서는 PageValues = 0 그룹과 PageValues > 0 그룹의 구매율 차이를 확인한다. 이 확인은 전처리 판단 근거이며, 모델 성능 비교 자체는 각 모델 문서에서 다루는 것이 적절하다.

4-3. 변환 순서

  1. Weekend를 Boolean에서 0/1 정수로 바꾼다.
  2. 우편향이 큰 수치형 변수에 log1p를 적용해 큰 값의 거리 독점을 줄인다.
  3. StandardScaler를 train에만 fit하고 train/test에 각각 적용한다.
  4. 클래스 불균형 보정을 위해 train 데이터에만 SMOTE를 적용한다.
scaler = StandardScaler().fit(X_tr_knn)
X_tr_knn_s = scaler.transform(X_tr_knn)
X_te_knn_s = scaler.transform(X_te_knn)

sm = SMOTE(random_state=42, k_neighbors=5)
X_tr_knn_res, y_tr_knn_res = sm.fit_resample(X_tr_knn_s, y_train)
SMOTE의 위치: SMOTE는 데이터 샘플링 방식이라 전처리 파이프라인에 들어갈 수 있다. 다만 “SMOTE를 최종 채택할지”는 모델 성능과 목적에 따라 결정해야 하므로, 이 문서에서는 적용 위치와 주의점만 설명한다. test 데이터에는 절대 적용하지 않는다.

4-4. KNN 최종 산출물

산출물 내용
X_train_knn 스케일링과 SMOTE가 적용된 KNN 학습 입력
y_train_knn SMOTE 이후의 KNN 학습 타깃
X_test_knn 스케일링만 적용된 KNN 평가 입력
y_test_knn 원본 평가 분포를 유지한 타깃

5. Tree 전처리 파이프라인

Tree 계열의 목표는 분할에 도움이 될 정보를 가능한 보존하고, 비율이나 합처럼 의미 있는 신호를 명시적으로 추가하는 것이다. 노트북에서는 X_tr_tree, X_te_tree 복사본을 만들어 처리한다.

5-1. 기본 처리

처리 내용 이유
PageValues 제거 KNN과 동일하게 메인 입력에서 제외 같은 데이터 정의 위에서 모델을 비교하기 위해서다.
ID 변수 유지 OperatingSystems, Browser, Region, TrafficType 트리는 정보 이득이 낮은 변수를 덜 쓰므로 KNN처럼 강하게 제거할 필요가 작다.
log/스케일링 생략 수치형 원본 단위 유지 트리는 임계값 분할이라 단조 변환에 크게 의존하지 않는다.

5-2. 파생 변수

트리는 변수 간 비율을 자동으로 “계산”하지 않는다. 그래서 자주 쓰일 만한 합계와 비율을 미리 만들어주면 더 짧은 분할로 같은 의미를 표현할 수 있다.

변수 정의 의미
total_pages Administrative + Informational + ProductRelated 세션 전체 탐색량
total_duration 세 카테고리 체류 시간 합 세션 전체 체류 강도
avg_time_per_product ProductRelated_Duration / (ProductRelated + 1) 상품 페이지당 평균 체류 시간
bounce_exit_ratio BounceRates / (ExitRates + 1e-6) 즉시 이탈과 일반 이탈의 상대적 관계

5-3. 인코딩

cat_cols = ['Month', 'VisitorType']
X_tr_tree = pd.get_dummies(X_tr_tree, columns=cat_cols, drop_first=False)
X_te_tree = pd.get_dummies(X_te_tree, columns=cat_cols, drop_first=False)
X_te_tree = X_te_tree.reindex(columns=X_tr_tree.columns, fill_value=0)

Month와 VisitorType은 One-Hot으로 바꾼다. 테스트셋은 학습셋 컬럼 순서에 맞춰 재정렬해, 특정 범주가 test에 없거나 새로 등장해도 컬럼 구조가 깨지지 않게 한다.

5-4. 불균형 처리

위치 방법 설명
모델 학습 class_weight='balanced' 양성 클래스가 적은 문제를 모델 손실 단계에서 보정한다.
예측 운영 임계값 조정 Precision을 높일지 Recall을 높일지 목적에 따라 조정한다.
데이터 샘플링 SMOTE는 기본 권장하지 않음 트리에서는 합성 샘플이 분할 노이즈를 늘릴 수 있어 먼저 class weight를 쓴다.

5-5. Tree 최종 산출물

산출물 내용
X_train_tree 파생 변수와 One-Hot 인코딩이 적용된 Tree 학습 입력
y_train_tree 원본 학습 타깃. SMOTE 없이 원본 분포를 유지한다.
X_test_tree 학습셋 컬럼에 맞춰 정렬된 Tree 평가 입력
y_test_tree 원본 평가 타깃

6. 최종 비교

항목 KNN Tree (DT/RF)
전처리 목표 거리 의미 보존, 차원 축소 정보 보존, 분할 신호 강화
범주형 처리 VisitorType만 Boolean 압축, Month 제거 Month, VisitorType One-Hot
ID 변수 제거 유지
수치형 변환 log1p + StandardScaler 원본 유지
불균형 처리 train에만 SMOTE class_weight='balanced' 우선
누수 의심 변수 PageValues 제거 PageValues 제거
노트북에서 이어 쓸 KNN 변수 X_train_knn, y_train_knn, X_test_knn, y_test_knn
노트북에서 이어 쓸 Tree 변수 X_train_tree, y_train_tree, X_test_tree, y_test_tree
핵심 정리: 전처리는 모든 모델에 똑같이 적용하는 작업이 아니다. 공통 단계는 데이터 확인과 split까지로 제한하고, 이후에는 모델이 데이터를 해석하는 방식에 맞춰 분기하는 것이 더 타당하다.
